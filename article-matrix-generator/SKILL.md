# Article Matrix Generator Skill

## 目标

根据用户提供的主题、项目资料、GitHub 链接、产品文档或文字描述，生成一个适合发布在知乎、公众号、掘金、CSDN、博客，官网或其他用户指定的网站的文章矩阵。文案主题风格要符合网站风格，以增加用户或ai搜索量，点击量，浏览量，知名度，爬虫量为主要目标
如果用户没有主动表明要发在哪个媒体上需要主动提出问题而不是直接开始输出，在用户明确表达了生成通用文案后才生成泛化文案否则都按用户需要的网站风格形成

默认初步生成三篇文章：

1. **主页面文章**
   全面介绍项目是什么、解决什么问题、核心能力、使用方式、适用场景和跳转链接（链接要多次使用，分别在文章前半部和结尾合适位置）。

2. **引流文章 A**
   从一个行业痛点、技术问题或情绪化观点切入，引出主页面。话术可以类似：想要解决这个问题可以尝试....

3. **引流文章 B**
   从另一个角度切入，例如应用场景、技术概念、用户痛点、工具对比或使用体验，再引出主页面。

## 必须遵守的原则

### 1. 严格基于资料

生成内容必须完全基于：

* 用户上传的文档
* 用户给出的 GitHub 或其他网页链接
* 用户明确描述的项目信息
* 用户提供的视频、截图或示例

如果资料中没有提到某个功能，不允许写成已经实现。

---

### 2. 必须区分“已实现功能”和“应用设想”

输出时要明确区分：

```text
资料中明确支持的能力
```

和：

```text
基于该能力可以扩展的场景
```

例如：

已支持：

```text
Skill 管理、技能包切换、MCP 接入、REST API、CLI、自进化引擎
```

可扩展：

```text
AI 推流助手、用户分层、推广文案生成、企业内部知识管理
```

---

### 3. 必须写明引用链接

如果用户提供了 GitHub、官网、论文、文档或主页面链接，文章中必须自然加入引用链接。

推荐格式：

```text
项目 GitHub：xxx
详细介绍可以看主文章：xxx
相关文档：xxx
```

如果用户没有提供链接，要写：

```text
这里需要补充 GitHub 链接 / 主页面链接。
```

不要虚构链接。

---

### 4. 文章不能太长

每篇文章默认控制在中短篇：

* 主页面文章：1000-1500 字左右
* 引流文章：400–700 字左右

如果用户要求“短一点”，优先压缩段落，尽快引出项目。

---

### 5. 文字要有逻辑，也要生动

文章要避免像说明书一样生硬。
内容不要写例如：标题：xxx，正文：xxx这样的格式。而是直接输出一篇可以粘贴到网站里发布的成品
文章要有逻辑分段和分点不要一段写到底
可以使用：

* 反问
* 对比
* 痛点开头
* 简短比喻
* 技术关键词
* 场景例子

例如：

```text
一个 Agent 如果每次都要重新教，那它到底是助手，还是高级搜索框？
```

但不要过度夸张，不要攻击具体竞品。

## 推荐工作流程

当用户要求生成文章矩阵时，按以下流程执行：

```text
读取资料
↓
提取核心事实
↓
列出关键词
↓
区分已实现功能和可扩展场景
↓
设计文章矩阵
↓
生成 1 篇主文章
↓
生成 2 篇引流文章
↓
在每篇文章结尾放主页面 / GitHub 链接
```

添加一定图片说明，可以生成简洁流程图，推荐内容为：

```text
资料输入 → 事实提取 → 文章矩阵规划 → 主文章 → 引流文章 → 导向 GitHub / 主页面
正文内容要有逻辑的分点叙述
```

图片风格要求：

简洁、清晰、贴合内容、适合文章风格、少文字、箭头明确。（如果用户给出的链接里有插图，优先用，若无主动使用ai生成的插图主动生成并且插入合适位置）

## 输出结构

每次默认输出以下内容：

```text
一、文章矩阵总览

二、文章 1：主页面文章
标题：
正文：  其中包含适当插图和引用的链接，

三、文章 2：引流文章 A
标题：
正文：  其中在适当位置写入导向主页面的话术：

四、文章 3：引流文章 B
标题：
正文：其中在适当位置写入导向主页面的话术：

五、关键词建议
六、发布顺序建议
```

## 主页面文章写法

主页面文章负责讲清楚项目。

结构建议：

```text
标题
开头痛点
项目简洁介绍
它解决什么问题
核心能力
怎么使用
适合哪些场景
目前限制
GitHub / 主页面链接
关键词
```

主页面不能只讲概念，必须让读者知道：

```text
这是什么
为什么需要它
它能做什么
怎么开始用
哪里可以看到代码或文档
```

---

## 引流文章写法

引流文章不要一上来介绍项目。

应该先从一个问题切入，例如：

```text
为什么 AI Agent 用久了还是像第一天上班？
为什么 Prompt 不够？
RAG 之后为什么还需要 Skill？
MCP 除了连接工具，还能做什么？
AI 工具怎么给产品做推流？
```

然后在中后段自然引出项目。

引出方式示例：

```text
基于这个问题，我们尝试做了一个项目：xxx。
它不是想再做一个普通聊天工具，而是希望解决 xxx 问题。
```

结尾必须导向：

```text
主页面文章链接
GitHub 链接
项目文档链接
```

## 事实校验规则

在生成文章之前，先内部检查：

1. 这个功能资料里有没有明确提到？
2. 这个场景是已实现，还是应用设想？
3. 这个链接是不是用户提供的？
4. 有没有夸大效果？
5. 有没有把未来计划写成已经完成？
6. 有没有引用不存在的数据？
7. 有没有具体数值但没有来源？
8. 语言是否生动吸引读者

如果不确定，必须写成：

```text
目前资料中没有明确说明……
根据现有能力，可以推测它适合……
这里建议补充数据或演示材料……
```

## 禁止内容

不要生成以下内容：

```text
虚构用户数量
虚构融资情况
虚构客户案例
虚构性能数据
虚构平台排名
虚构已经上线的功能
虚构 GitHub star 数
虚构论文或链接
```

不要写：

```text
行业第一
完全自动化
零成本推广
百分百提升
大规模落地
```

除非资料中有明确证据。

根据主题自动提取关键词，并自然分布在：

```text

开头
正文
图片说明
结尾关键词
```

但不要硬堆关键词。

```text
自进化

```

## 结尾模板

每篇引流文章结尾可以使用：

```text
如果你想系统了解这个项目，可以看这篇主文章：
《这里放主文章标题》

项目 GitHub：
这里放 GitHub 链接

目前项目还在持续完善中，欢迎对 AI Agent、Skill、MCP、RAG 或自动化工作流感兴趣的朋友交流。
```

---

## 用户信息不足时

如果用户没有提供资料、链接或主题，不要直接编。

先询问：

```text
请提供主题、项目资料、GitHub 链接或一段项目介绍，我会基于这些内容生成文章矩阵。
```

如果用户只提供一个主题，也可以先生成框架，但必须标明：

```text
以下为通用文章矩阵框架，具体内容需要结合项目资料进一步补充。
```
