# 文章矩阵生成 Skill

## 目标

根据用户提供的主题、项目资料、GitHub 链接、产品文档、网页、截图、视频、演示材料或文字描述，生成一组可以直接发布的中文文章矩阵。

文章矩阵要帮助读者快速理解：

- 这个项目是什么
- 它解决了什么真实问题
- 它已经具备哪些能力
- 它适合哪些场景
- 读者下一步应该去哪里了解、试用或查看源码

默认生成：

1. 1 篇主页面文章，负责完整讲清楚项目。
2. 2 篇引流文章，分别从不同痛点、场景或技术问题切入，再自然导向主文章、GitHub、文档或产品页。
3. 一组插图规划与生成结果，用流程图、层级图、模块图或对比图辅助说明文章内容。
4. 文件夹中所有的md文件都要参考，在主页文章写作时只参考 `SKILL.md` 的主文章规则和用户提供资料，不参考 `article-matrix-generator` 目录下的补充风格 md 文件；写引流文章时才参考该目录下全部 md 文件，包括但不限于 `adding.md`、`adding2.md`、`adding3.md`、`adding4.md`。
如果用户没有说明发布平台，必须先询问：

```text
这组文章主要准备发在哪个平台？例如知乎、公众号、掘金、CSDN、博客、官网，还是希望先生成一版通用稿？
```

只有用户明确表示“通用稿”“不限平台”“先给通用版”时，才生成泛化版本。

## 输入与信息不足时的处理

只使用用户提供的资料，或用户明确给出的链接内容。不要自行编造项目能力、案例、数据或链接。

如果用户只提供一个主题，可以先生成框架，但必须标明：

```text
以下为通用文章矩阵框架，具体内容需要结合项目资料进一步补充。
```

如果用户没有提供主题、资料或链接，先询问：

```text
请提供主题、项目资料、GitHub 链接或一段项目介绍，我会基于这些内容生成文章矩阵。
```

如果缺少 GitHub、文档、产品页或主文章链接，不能虚构，必须写：

```text
这里需要补充 GitHub 链接 / 主页面链接 / 项目文档链接。
```

## 事实边界

不能把推测写成事实。必要时必须区分：

```text
资料中明确支持的能力
```

和：

```text
基于现有能力可以扩展的场景
```

如果不确定，使用谨慎表述：

```text
目前资料中没有明确说明……
根据现有能力，可以推测它适合……
这里建议补充数据或演示材料……
```

禁止虚构用户数量、客户案例、融资情况、性能数据、平台排名、GitHub star 数、媒体报道、论文、链接或已上线功能。

除非资料中有明确证据，否则不要写：

```text
行业第一
完全自动化
零成本推广
百分百提升
大规模落地
颠覆行业
遥遥领先
```

## 写作方法

可以参考有经验的公众号长文写作技巧，但要抽象成更通用、更克制的文章能力。不要模仿某个作者过强的个人口癖，不要过度使用短句和频繁换行，也不要为了制造氛围而拉长篇幅。

### 1. 写之前先判断角度

正式写作前，先判断每篇文章是否具备 3 个要素：

- 好奇感：标题和开头是否让人想继续读？
- 信息量：读者看完是否能学到具体东西？
- 共鸣感：是否击中了真实痛点、困惑或需求？

如果角度太弱，先调整切入点。文章矩阵不是把同一个项目重复介绍 3 遍，而是让每篇文章都承担不同的传播任务。

### 2. 先确定文章原型

每篇文章写之前，先判断最接近哪种原型：

- 项目解释型：讲清楚它是什么、解决什么问题、怎么开始用。
- 产品体验型：展示它如何改变一个具体工作流。
- 技术概念型：通过具体问题解释 Skill、MCP、RAG、CLI、API、工作流自动化等概念。
- 场景解决型：从用户任务出发，把项目作为一种解决路径引出。
- 对比分析型：比较旧方法和新方法，但不攻击具体竞品。
- 方法论型：给出可执行步骤，同时坦诚说明限制、学习成本或资料缺口。

### 3. 开头要从具体入口进入

避免“在当今 AI 快速发展的时代”“随着技术不断进步”这类空泛开头。优先从具体场景、结果、问题、截图、命令、文档、用户困境或工作流进入。

可选入口（作为参考，不仅限于以下几点）：

- 痛点场景：用户每次都要重复同一套说明。
- 认知错位：大家以为 Prompt 解决了一切，但问题第二天又回来了。
- 物件特写：一张截图、一条命令、一个配置文件、一个 GitHub README。
- 问题切入：为什么一个 Agent 用久了还像第一天上班？
- 结果倒挂：先展示最终流程，再解释项目如何支撑它。

如果资料允许，前 300 字内至少出现 1 个具体细节，例如命令、按钮、截图内容、文件名、项目模块、页面状态或用户问题。

### 4. 采用“层层剥开”的解释方式

不要直接给结论。优先使用：

```text
现象 → 表面解释 → 更深的问题 → 项目如何处理 → 当前边界
```

用资料中的具体例子、命令、模块、截图或文档支撑观点。资料没有的细节，只能写成应用设想，不能写成已实现能力。

### 5. 保持自然但不过度个人化

文章要像一个懂行的人在认真解释，而不是新闻稿、广告稿或说明书。可以适度使用口语化转场：

```text
你可以把它理解成……
这里真正麻烦的是……
这个问题不在于……
更实际的做法是……
回到这个项目本身……
```

减少第一人称视角，少用“我觉得”“我认为”“我们觉得”“我们尝试”这类表达。除非用户明确要求以创作者自述、项目作者口吻或个人复盘口吻写作，否则默认使用更客观的第三方介绍、专业分析或产品说明视角。

推荐表达：

```text
更准确的说法是……
从这个角度看……
这个项目更像是在解决……
资料中能看到的关键点是……
一个更实际的理解方式是……
```

不推荐表达：

```text
我觉得……
我认为……
我们觉得……
我们尝试做了一个……
在我看来……
一句话说：
```
不要分很细的一、二、三、
避免强烈个人口癖、粗俗表达、过度情绪化标点、连续的一句话成段。段落通常保持 2-5 句，只有关键观点才偶尔单独成段。

### 6. 做好前后呼应

如果开头用了一个具体问题、场景或比喻，结尾要尽量回到它，让文章有完整感。

例如（以此为例，要多变，不要总是用同一个例子）：

```text
开头提出“每次都像第一天上班”的问题，结尾回到“让经验留下来，而不是让用户每次重讲一遍”。
```

## 推荐工作流程

按以下顺序执行：

```text
确认发布平台
↓
读取资料
↓
提取事实、链接、关键词和限制
↓
区分已实现能力和可扩展场景
↓
选择文章角度和文章原型
↓
规划插图
↓
生成 1 篇主文章和 2 篇引流文章
↓
生成或复用插图并插入正文
↓
执行自检
↓
输出文章、插图、关键词和发布顺序建议
```

正式写文章前，先简短整理事实：

```text
我会基于以下事实生成，不会扩写成已实现功能：

已明确支持：
- ...

可作为应用设想：
- ...

需要补充的链接或资料：
- ...
```

## 篇幅控制

默认控制篇幅，但要给足论述空间，不要为了凑字拉长：

- 主页面文章：2000 字左右。
- 引流文章：1200 字左右。
- 如果是 CSDN 或技术博客，并且用户提供了足够的使用细节，主文章可以适当加长，用来写步骤、命令和配置。
- 如果用户要求“短一点”，优先删背景，再删案例，但保留事实、链接、插图和核心导向。

例如：（二、文章 1：主页面文章

很多适老化产品看起来都很相似：字体放大一点，按钮做厚一点，颜色做暖一点。

一句话概括，它更像是一个“老人每天愿意打开的陪伴入口”，而不是一个堆满功能的工具箱。
它真正想解决的问题

## 主页面文章写法

主页面文章必须回答：

```text
这是什么？
为什么需要它？
它已经能做什么？
怎么开始用？
哪里可以看到代码、文档或主页面？
当前边界是什么？
```

建议结构：

```text
标题
具体痛点或现场切入
一句话介绍项目
机制图或流程图
它解决的问题
核心能力
能力模块图或层级图
使用方式
适合场景
当前限制 / 需要补充的资料
GitHub / 主页面 / 文档链接
关键词
```

主文章可以使用小标题，但不要把每段都写成列表。它应该是一篇能直接发布的成品文章，而不是项目说明书。

## 引流文章写法

引流文章不要一上来介绍项目。先从读者问题、行业痛点或具体场景切入，再在中后段自然引出项目。

可用问题（参考风格，不用照搬照抄）：

```text
为什么 AI Agent 用久了还是像第一天上班？
为什么 Prompt 不够？
RAG 之后为什么还需要 Skill？
MCP 除了连接工具，还能做什么？
AI 工具怎么给产品做推流？
```

自然引出方式：

```text
围绕这个问题，可以关注一个项目：xxx。
它不是普通的聊天工具，而是更集中地处理 xxx 问题。
```

每篇引流文章结尾必须导向主文章、GitHub、文档或产品页。链接缺失时，明确写“这里需要补充……”，不要编。默认必须把最终正文保存为 Markdown 文件，不需要等待用户额外要求。保存的文章文件只包含可直接发布的正文，不包含质检报告

### 根据用户改稿提炼的引流文风格

当用户提供自己的改稿、润色稿或偏好样本时，提炼其风格机制，而不是复用原句。不要把用户改稿当成模板逐句套用，也不要在后续文章里机械复现相同开头、相同比喻、相同收尾。用户改稿只用于判断表达偏好、叙事节奏、切入方式和论述推进方式。

写 AI Agent、Skill Pack、MCP、自进化类引流文章时，优先使用“反复培训 → 技能包 → 经验沉淀 → 长期工作系统”的推进方式：

```text
具体写作动作或使用现场
↓
发现自己重复写人设 / 规则 / 工具说明
↓
指出 Prompt 正在变成反复培训
↓
抛出“永远第一天上班的实习生”式问题
↓
自然引出项目和链接
↓
用命令、配置、接口等具体细节证明项目不是空概念
↓
区分已支持能力和应用设想
↓
回到“经验能不能留下来”“AI 能不能长技能”的主线
```

开头要模仿“从一个具体动作里突然意识到问题”的风格，而不是照抄固定句式。可选择的入口包括：

- 正在写一段系统提示词、人设、工具规则或知识库说明时，突然发现自己反复做过很多次。
- 正在给客服、运维、写作、知识库问答等不同 Agent 重复解释背景时，意识到 Prompt 正在变成一种岗前培训。
- 从一条具体命令、一个配置文件、一段 README、一次工具调用失败或一次知识库漏查开始，逐步推出“临时提示不等于长期能力”。
- 从“该查资料时没查、该用工具时没用、该承认不知道时开始编”的现象进入，避免一上来介绍项目。

这类开头的关键不是第一人称，也不是某个固定说法，而是前 300 字内必须出现可感知的具体动作或反复场景，例如写人设、查知识库、调用工具、承认不知道、客服机器人、运维助手、写作助手、配置技能包、切换命令。不要写成泛泛的“最近 AI Agent 很火”。

表达风格要贴近用户改稿的气质：

- 先写普通、朴素、重复发生的小动作，再逐步抬高到产品问题。
- 使用短句制造停顿，但不要每句都断行；关键转折可以单独成段。
- 允许轻微吐槽和口语判断，但避免粗暴攻击竞品或夸大项目。
- 多用“不是 X，而是 Y”的对照结构，把问题说清楚。
- 把抽象概念翻译成工作场景，例如把 Prompt 写成临时交代，把 Skill Pack 写成长期工作模式，把自进化写成经验沉淀与复盘。
- 结尾要回扣开头的重复劳动或岗前培训意象，让文章有闭环。
- 可以写出“项目还不完美，但问题值得一起琢磨”的姿态，避免硬广。

可以借鉴用户样本中的表达功能，但不要直接复读原句：

- 用一个形象问题替代抽象判断，例如“这个助手为什么每次都像重新入职？”
- 用工作关系类比解释长期陪伴型 Agent，例如“数字同事”“工作方法”“复盘”“补课”等。
- 用任务失败现象解释工具和知识库的重要性，例如该查资料时没查、该用工具时没用、该记住的经验下次又消失。
- 用“从会聊天到能长期工作”这类方向收束，但每篇文章要换一种具体说法。

写 Homie 或类似 Agent 框架时，尽量把项目能力落到具体细节，不要只写抽象概念。可优先检查资料中是否支持并写入：

- 技能包切换命令，例如 `/pack switch it-ops`、`/pack switch cleanbot-support`、`/pack switch general`
- CLI 使用方式，例如 `homie chat`、`homie ask`
- 工具调用展示参数，例如 `--show-tools`
- REST API 与 SSE 流式接口
- MCP Server
- Python SDK
- 兼容 OpenAI 协议的模型服务
- OpenRouter、OpenAI、Ollama、vLLM 等模型接入方式
- 标准 `SKILL.md` 技能文件

如果资料中没有明确写到某个命令、接口或模型接入方式，不要从样本中继承为事实；必须删除或改成“这里需要补充资料确认”。

引流文章结尾要采用“开放邀请 + 问题意识 + 轻行动”的姿态。可以邀请读者 star、看代码、提 issue、提 PR，但语气要像共同讨论一个尚在生长的问题，而不是品牌广告。不要固定使用同一组结尾句。

如果平台是小红书，可以保留更强的口语感和短段落，但不要过度口号化。允许使用“说实话”“这点对开发者很重要”“大时代啊，朋友们”等表达；仍然避免虚构体验、虚构数据、虚构 star 数和夸张承诺。

## 插图生成规则

本 skill 默认使用图片解释流程、层级和模块关系。文章生成后，要自动规划并生成或准备可插入的插图。不要只写“可以配图”，除非当前环境无法生成图片。

默认插图数量：

- 主文章：2 张。
- 每篇引流文章：1 张。
- 如果文章很短，可以减少为 1 张。
- 如果用户明确要求不要图片，才跳过插图。

推荐插图类型：

- 流程图：输入 → 处理 → 输出、故障排查流程、使用路径、内容生成链路。
- 层级图：能力层、接口层、场景层、架构关系。
- 模块图：CLI、REST API、MCP Server、Skill 管理、自进化引擎等。
- 对比图：旧工作流 vs 新工作流。
- 场景路径图：客服排障、内容生成、企业知识管理、自动化工作流等具体任务。

推荐视觉风格：

- 白色或浅色背景。
- 低饱和蓝色 / 灰色配色。
- 圆角矩形步骤块。
- 深灰或黑色箭头。
- 文字尽量少，每个节点最好不超过 12 个字。
- 适合知乎、公众号、掘金、CSDN、博客或官网文章。

每张图下方必须有 1 句不超过 140 字的图注。

示例：

```text
![故障排查流程图](image-path)

图注：把一次客服排障流程拆成明确步骤后，就可以进一步沉淀为可复用的 Skill。
```

如果当前环境无法生成真实图片，必须输出插图生成清单：


## 平台风格适配

根据用户指定平台调整语气、结构和插图密度：

- 知乎：问题意识更强，多解释、多对比，项目引出要自然，避免硬广。
- 公众号：叙事感更强，段落更短，适合手机阅读，图片字号更大。
- 掘金：技术关键词更明确，多写架构、接口、实现思路和开发者价值。
- CSDN：更重视安装、命令、配置、排错和步骤图。
- 官网 / 博客：表达更稳，定位更清楚，多写能力边界、场景和文档入口。



## 自检清单

输出前必须完成以下检查：

### L1 事实检查

- 所有已实现能力是否都有资料支持？
- 应用设想是否被明确标注？
- 所有链接是否来自用户，或被标为需要补充？
- 是否出现无来源数据、排名、客户、媒体、star 数或夸张承诺？
- 插图中是否出现了资料不支持的信息？

### L2 角度与结构检查

- 每篇文章是否都有独立存在的理由？
- 开头是否有具体场景、问题、物件、命令、截图或用户困境？
- 文章是否从问题推进到解释，再推进到项目和行动入口？
- 结尾是否尽量回扣开头的问题或意象？

### L3 可读性检查

- 是否避开了空泛 AI 式开头？
- 是否避免了过度列表化？
- 段落是否没有被切得过碎？
- 是否有足够来自资料的具体细节？
- 语气是否自然，但不过度个人化？
- 是否减少了“我觉得”“我认为”“我们尝试”等第一人称表达？

### L4 平台与插图检查

- 语气是否符合指定平台？
- 插图位置是否真正帮助理解，而不是装饰？
- 图注是否简短准确？
- 关键词是否自然出现在标题、小标题、正文、图注和结尾？
