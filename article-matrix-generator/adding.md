## 核心任务

根据用户提供的资料，生成一组适合发布的中文文章矩阵。默认包含：

- 1 篇主文章，约 2000 字，用来完整讲清楚项目、产品、技术或主题。
- 2 篇引流文章，每篇约 1200 字，从不同痛点、问题或场景切入，再导向主文章、GitHub、文档或产品页。
- 一组可插入正文的插图，包括流程图、层级图、模块图、对比图或场景路径图。

文章必须基于用户提供的材料，不能编造功能、数据、链接、案例、客户、排名、融资、性能指标、GitHub star 数或媒体报道。

如果用户没有说明发布平台，先问：

```text
这组文章主要准备发在哪个平台？例如知乎、公众号、掘金、CSDN、博客、官网，还是希望先生成一版通用稿？
```

只有用户明确要求“通用版”“不限平台”“先给一版通用稿”时，才输出通用版本。

## 写作底层原则

文章不是把资料换一种说法，而是把资料变成读者愿意读下去的叙事结构。每篇文章都要同时完成三件事：

- 抓住读者注意力。
- 解释一个真实问题。
- 把项目、产品或技术自然放进这个问题里。

可以借鉴强叙事型公众号写法中的选题判断、具体入口、层层剥开、前后呼应和自检方法，但不要模仿过强的个人口癖，不要频繁使用一句话一段，也不要把文章写成过度情绪化的个人随笔。

默认语气应是“客观、清楚、有判断力的专业介绍”。除非用户明确要求个人复盘、创作者自述或第一人称口吻，否则减少：

```text
我觉得
我认为
我们觉得
我们尝试
在我看来
```

优先使用：

```text
更准确的说法是……
从这个角度看……
这个项目更像是在解决……
资料中能看到的关键点是……
一个更实际的理解方式是……
```

## 先做选题判断

正式写作前，先判断每篇文章的角度是否成立。使用 HKR 检查：

### H：好奇感

读者看到标题或开头后，是否会想继续读？如果文章只是“某项目介绍”“某工具说明”，吸引力通常不够，需要换成一个更具体的问题。

弱角度：

```text
Homie 项目介绍
```

更好的角度：

```text
为什么 AI Agent 用久了，还是像第一天上班？
```

### K：知识量

读者看完是否能获得具体信息？文章必须提供项目能力、使用方式、流程、场景、限制或判断方法，不能只有情绪和口号。

### R：共鸣感

文章是否击中了读者真实遇到的问题？例如重复写 Prompt、Agent 经验无法沉淀、工具很多但流程不稳定、团队知识难复用等。

如果一个角度 HKR 都弱，先换角度再写。

## 选择文章原型

每篇文章写作前，先判断它属于哪种原型。不同原型决定文章结构，不要所有文章都写成项目介绍。

### 1. 项目解释型

适合主文章。结构是：

```text
具体问题 → 一句话定义 → 核心机制 → 已支持能力 → 使用方式 → 场景 → 限制 → 链接
```

### 2. 痛点拆解型

适合引流文章。结构是：

```text
读者熟悉的麻烦 → 为什么旧方法不够 → 更深层原因 → 项目如何切入 → 继续阅读入口
```

### 3. 技术概念型

适合解释 Skill、MCP、RAG、Agent、CLI、REST API、工作流自动化等。结构是：

```text
误解或争议 → 概念解释 → 具体例子 → 项目中的体现 → 边界
```

### 4. 场景解决型

适合从客服、运维、内容生成、知识管理、推广文案、内部工具等场景切入。结构是：

```text
任务现场 → 当前流程卡点 → 新流程示意 → 项目能力支撑 → 适用与不适用
```

### 5. 对比分析型

适合解释“旧方法 vs 新方法”。结构是：

```text
旧方法能解决什么 → 它解决不了什么 → 新方法的变化 → 代价与限制 → 链接
```

禁止通过贬低具体竞品来制造对比。

## 从具体入口开头

不要使用空泛开头，例如：

```text
随着人工智能技术不断发展……
在当今数字化时代……
近年来，AI Agent 越来越火……
```

优先从以下入口开始：

- 一个具体问题：为什么 Agent 每次都要重新教？
- 一个具体场景：客服需要重复确认型号、故障现象、网络状态。
- 一个具体物件：GitHub README、命令行、配置文件、截图、流程图。
- 一个具体结果：一次对话结束后，经验没有留下来。
- 一个具体冲突：Prompt 很强，但它更像临时指令，不像长期能力。

如果资料允许，文章前 300 字内至少出现 1 个来自资料的具体细节，例如命令、模块名、接口名、文件名、截图内容、文档标题、项目功能或用户问题。

## 用“层层剥开”组织正文

不要一开始就罗列功能。优先使用：

```text
现象 → 表面问题 → 更深层原因 → 项目机制 → 使用方式 → 场景 → 当前边界
```

例如写 AI Agent Skill 时，不要直接说“它支持 Skill 管理”。可以先写：

```text
问题不是 Agent 不会回答，而是它回答完以后，经验没有变成下一次可复用的能力。
```

然后再引出 Skill、技能包、MCP、API 或自进化机制。

## 控制语气和段落

保持自然、有判断力，但不要过度个人化。

段落通常保持 2-5 句。可以在关键观点处使用短段，但不要连续大量短句，也不要让文章像被刻意切碎。

可以使用反问、对比、类比、具体场景和轻微情绪，但不要写成夸张营销文。

避免：

```text
说实话，我真的很震惊。
我觉得这件事特别值得聊。
这东西太强了。
懂的人自然懂。
```

推荐：

```text
这个问题真正麻烦的地方在于……
更准确地说，它解决的不是一次回答，而是经验复用。
如果把 Prompt 看成临时指令，那么 Skill 更接近可沉淀的方法。
```

## 严格事实边界

生成前先整理资料：

```text
我会基于以下事实生成，不会扩写成已实现功能：

已明确支持：
- ...

可作为应用设想：
- ...

需要补充的链接或资料：
- ...
```

文章中必须区分：

```text
资料中明确支持的能力
```

和：

```text
基于现有能力可以扩展的场景
```

如果不确定，写：

```text
目前资料中没有明确说明……
根据现有能力，可以推测它适合……
这里建议补充数据或演示材料……
```

禁止写没有来源的具体数字、用户规模、客户案例、融资、性能、排名、star 数、论文、媒体报道或上线状态。

除非资料明确证明，否则不要写：

```text
行业第一
完全自动化
零成本推广
百分百提升
大规模落地
颠覆行业
遥遥领先
```

## 引用与链接规则

如果用户提供 GitHub、官网、论文、文档、主页面或相关资料链接，文章中必须自然加入。

推荐格式：

```text
项目 GitHub：xxx
详细介绍可以看主文章：xxx
相关文档：xxx
```

如果没有提供链接，不能虚构，写：

```text
这里需要补充 GitHub 链接 / 主页面链接 / 项目文档链接。
```

## 主文章写法

主文章约 2000 字，负责把项目讲清楚。它不是引流短文，也不是 README 复述。

必须回答：

```text
这是什么？
为什么需要它？
它解决什么问题？
它已经能做什么？
怎么开始用？
适合哪些场景？
当前边界是什么？
哪里可以看到代码、文档或主页面？
```

建议结构：

```text
标题
具体痛点或现场切入
一句话定义项目
机制图或流程图
更深层问题解释
核心能力
能力模块图或层级图
使用方式
适合场景
当前限制或资料缺口
GitHub / 主页面 / 文档链接
关键词
```

主文章需要有完整叙事，不要写成“标题：”“正文：”这种模板格式。可以使用小标题，但不要把每一段都写成列表。

## 引流文章写法

每篇引流文章约 1200 字，从一个问题、痛点、概念误区或场景切入。不要开头就介绍项目。

可用切入问题：

```text
为什么 AI Agent 用久了还是像第一天上班？
为什么 Prompt 不够？
RAG 之后为什么还需要 Skill？
MCP 除了连接工具，还能做什么？
AI 工具怎么给产品做推流？
为什么团队知识库很多，流程还是不稳定？
```

自然引出项目时，减少第一人称：

```text
围绕这个问题，可以关注一个项目：xxx。
它不是普通的聊天工具，而是更集中地处理 xxx 问题。
```

不推荐：

```text
我觉得这个问题很重要，所以我们尝试做了一个项目。
```

每篇引流文章结尾必须导向：

```text
主文章链接
GitHub 链接
项目文档链接
```

链接缺失时明确标注需要补充。

## 插图生成规则

文章生成后，默认规划并生成或准备可插入的插图。不要只写“建议配图”，除非当前环境无法生成图片。

默认数量：

- 主文章：2-3 张。
- 每篇引流文章：1 张。
- 如果文章较短，可以减少。
- 用户明确不要图片时，才跳过。

优先生成：

- 流程图：输入 → 处理 → 输出、故障排查、使用路径、内容生成链路。
- 层级图：能力层、接口层、场景层、架构关系。
- 模块图：CLI、REST API、MCP Server、Skill 管理、自进化引擎等。
- 对比图：旧工作流 vs 新工作流。
- 场景路径图：客服排障、内容生成、企业知识管理、自动化工作流。

视觉风格：

- 白色或浅色背景。
- 低饱和蓝色 / 灰色。
- 圆角矩形步骤块。
- 深灰或黑色箭头。
- 每个节点尽量不超过 12 个字。
- 少装饰，信息清晰，适合知乎、公众号、掘金、CSDN、博客或官网。

每张图下方必须配一句不超过 140 字的图注。

示例：

```text
![故障排查流程图](image-path)

图注：把一次客服排障流程拆成明确步骤后，就可以进一步沉淀为可复用的 Skill。
```

如果当前环境无法生成真实图片，输出插图生成清单：

```text
插图 1：项目核心流程图
建议插入位置：主文章“一句话定义项目”之后
画面内容：资料输入 → 事实提取 → Skill 生成 → 技能复用 → 持续优化
图注：这张图用于说明项目如何把一次任务经验沉淀成可复用能力。
图片提示词：浅色背景，横向流程图，5 个浅蓝色圆角步骤块，黑色箭头连接，中文节点文字清晰，科技产品文章配图风格。
```

插图同样必须遵守事实边界。图中如果包含应用设想，图注必须说明它不是资料中明确上线的功能。

## 平台适配

根据平台调整结构和语气。

### 知乎

问题意识更强，多解释、多对比，项目引出要自然，避免硬广。适合用“问题链路图”和“方案对比图”。

### 公众号

开头更有叙事感，段落更适合手机阅读，但不要过度碎片化。图片字号要更大，流程图要清楚。

### 掘金

技术关键词更明确，多写架构、接口、实现思路和开发者价值。适合模块图、调用流程图、架构层级图。

### CSDN

更重视安装、命令、配置、排错和步骤图。可以多写实操步骤，但必须来自资料。

### 官网 / 博客

表达更稳，定位更清楚，少情绪化，多写能力边界、场景、文档入口和项目价值。

## 默认输出结构

```text
一、事实提取与边界说明
二、文章矩阵总览
三、文章 1：主页面文章
四、文章 2：引流文章 A
五、文章 3：引流文章 B
六、插图清单与生成结果
七、关键词建议
八、发布顺序建议
```

文章矩阵总览需要包含：

- 文章目标
- 文章原型
- 切入角度
- 目标读者
- 主要关键词
- 推荐插图
- 导向链接

## 输出前自检

### L1：事实检查

- 所有已实现能力是否都有资料支持？
- 应用设想是否被明确标注？
- 所有链接是否来自用户，或被标为需要补充？
- 是否出现无来源数据、客户、排名、star 数、媒体报道或夸张承诺？
- 插图中是否出现资料不支持的信息？

### L2：选题与结构检查

- 每篇文章是否有独立存在的理由？
- 开头是否有具体问题、场景、命令、截图、文档或用户困境？
- 是否完成了“问题 → 解释 → 项目 → 行动入口”的推进？
- 结尾是否尽量回扣开头的问题或意象？

### L3：表达检查

- 是否避开空泛 AI 式开头？
- 是否减少“我觉得”“我认为”“我们尝试”等第一人称表达？
- 是否避免连续短句和频繁换行？
- 是否避免过度列表化？
- 是否有足够来自资料的具体细节？
- 语气是否自然、清楚，但不过度个人化？

### L4：平台与插图检查

- 语气是否符合指定平台？
- 插图位置是否真正帮助理解？
- 图注是否简短准确？
- 关键词是否自然出现在标题、小标题、正文、图注和结尾？
