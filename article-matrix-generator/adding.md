## 写作底层原则

文章不是把资料换一种说法，而是把资料变成读者愿意读下去的叙事结构。每篇文章都要同时完成三件事：

- 抓住读者注意力。
- 解释一个真实问题。
- 把项目、产品或技术自然放进这个问题里。

可以借鉴强叙事型公众号写法中的选题判断、具体入口、层层剥开、前后呼应和自检方法，但不要模仿过强的个人口癖，不要频繁使用一句话一段，也不要把文章写成过度情绪化的个人随笔。

## 控制语气和段落

保持自然、有判断力，但不要过度个人化。

段落通常保持 2-5 句。可以在关键观点处使用短段，但不要连续大量短句，也不要让文章像被刻意切碎。

可以使用反问、对比、类比、具体场景和轻微情绪，但不要写成夸张营销文。

避免：

```text
说实话，我真的很震惊。
我觉得这件事特别值得聊。
这东西太强了。
懂的人自然懂。
```

推荐：

```text
这个问题真正麻烦的地方在于……
更准确地说，它解决的不是一次回答，而是经验复用。
如果把 Prompt 看成临时指令，那么 Skill 更接近可沉淀的方法。
```

## 严格事实边界

生成前先整理资料：

```text
我会基于以下事实生成，不会扩写成已实现功能：

已明确支持：
- ...

可作为应用设想：
- ...

需要补充的链接或资料：
- ...
```

文章中必须区分：

```text
资料中明确支持的能力
```

和：

```text
基于现有能力可以扩展的场景
```

如果不确定，写：

```text
目前资料中没有明确说明……
根据现有能力，可以推测它适合……
这里建议补充数据或演示材料……
```

禁止写没有来源的具体数字、用户规模、客户案例、融资、性能、排名、star 数、论文、媒体报道或上线状态。

除非资料明确证明，否则不要写：

```text
行业第一
完全自动化
零成本推广
百分百提升
大规模落地
颠覆行业
遥遥领先
```

## 引用与链接规则

如果用户提供 GitHub、官网、论文、文档、主页面或相关资料链接，文章中必须自然加入。

推荐格式：

```text
项目 GitHub：xxx
详细介绍可以看主文章：xxx
相关文档：xxx
```

如果没有提供链接，不能虚构，写：

```text
这里需要补充 GitHub 链接 / 主页面链接 / 项目文档链接。
```

## 引流文章写法

每篇引流文章约 1200 字，从一个问题、痛点、或场景切入。不要开头就介绍项目。

可用切入问题：

```text
为什么 AI Agent 用久了还是像第一天上班？
为什么 Prompt 不够？
RAG 之后为什么还需要 Skill？
MCP 除了连接工具，还能做什么？
AI 工具怎么给产品做推流？
为什么团队知识库很多，流程还是不稳定？
```

自然引出项目时，减少第一人称：

```text
围绕这个问题，可以关注一个项目：xxx。
它不是普通的聊天工具，而是更集中地处理 xxx 问题。
```

不推荐：

```text
我觉得这个问题很重要，所以我们尝试做了一个项目。
```

每篇引流文章结尾必须导向：

```text
主文章链接
GitHub 链接
项目文档链接
```

链接缺失时明确标注需要补充。

## 插图生成规则

文章生成后，默认规划并生成或准备可插入的插图。不要只写“建议配图”，除非当前环境无法生成图片。

默认数量：

- 主文章：2-3 张。
- 每篇引流文章：1 张。
- 如果文章较短，可以减少。
- 用户明确不要图片时，才跳过。

优先生成：

- 流程图：输入 → 处理 → 输出、故障排查、使用路径、内容生成链路。
- 层级图：能力层、接口层、场景层、架构关系。
- 模块图：CLI、REST API、MCP Server、Skill 管理、自进化引擎等。
- 对比图：旧工作流 vs 新工作流。
- 场景路径图：客服排障、内容生成、企业知识管理、自动化工作流。

视觉风格：

- 白色或浅色背景。
- 低饱和蓝色 / 灰色。
- 圆角矩形步骤块。
- 深灰或黑色箭头。
- 每个节点尽量不超过 12 个字。
- 少装饰，信息清晰，适合知乎、公众号、掘金、CSDN、博客或官网。

每张图下方必须配一句不超过 140 字的图注。

示例：

```text
![故障排查流程图](image-path)

图注：把一次客服排障流程拆成明确步骤后，就可以进一步沉淀为可复用的 Skill。
```

如果当前环境无法生成真实图片，输出插图生成清单：

```text
插图 1：项目核心流程图
建议插入位置：主文章“一句话定义项目”之后
画面内容：资料输入 → 事实提取 → Skill 生成 → 技能复用 → 持续优化
图注：这张图用于说明项目如何把一次任务经验沉淀成可复用能力。
图片提示词：浅色背景，横向流程图，5 个浅蓝色圆角步骤块，黑色箭头连接，中文节点文字清晰，科技产品文章配图风格。
```

插图同样必须遵守事实边界。图中如果包含应用设想，图注必须说明它不是资料中明确上线的功能。

## 平台适配

根据平台调整结构和语气。

### 知乎

问题意识更强，多解释、多对比，项目引出要自然，避免硬广。适合用“问题链路图”和“方案对比图”。

### 公众号

开头更有叙事感，段落更适合手机阅读，但不要过度碎片化。图片字号要更大，流程图要清楚。

### 掘金

技术关键词更明确，多写架构、接口、实现思路和开发者价值。适合模块图、调用流程图、架构层级图。

### CSDN

更重视安装、命令、配置、排错和步骤图。可以多写实操步骤，但必须来自资料。

### 官网 / 博客

表达更稳，定位更清楚，少情绪化，多写能力边界、场景、文档入口和项目价值。


## 输出前自检

### L1：事实检查

- 所有已实现能力是否都有资料支持？
- 应用设想是否被明确标注？
- 所有链接是否来自用户，或被标为需要补充？
- 是否出现无来源数据、客户、排名、star 数、媒体报道或夸张承诺？
- 插图中是否出现资料不支持的信息？

### L2：选题与结构检查

- 每篇文章是否有独立存在的理由？
- 开头是否有具体问题、场景、命令、截图、文档或用户困境？
- 是否完成了“问题 → 解释 → 项目 → 行动入口”的推进？
- 结尾是否尽量回扣开头的问题或意象？

### L3：表达检查

- 是否避开空泛 AI 式开头？
- 是否减少“我觉得”“我认为”“我们尝试”等第一人称表达？
- 是否避免连续短句和频繁换行？
- 是否避免过度列表化？
- 是否有足够来自资料的具体细节？
- 语气是否自然、清楚，但不过度个人化？

### L4：平台与插图检查

- 语气是否符合指定平台？
- 插图位置是否真正帮助理解？
- 图注是否简短准确？
- 关键词是否自然出现在标题、小标题、正文、图注和结尾？

## 关键词处理

根据主题自动提取关键词，并自然分布在标题、小标题、开头、正文、图片说明、图注和结尾关键词中。不要硬堆关键词。

如果主题是 AI Agent，可优先考虑：

```text
AI Agent
智能体
Skill
技能包
MCP
RAG
LLM
知识库
工具调用
REST API
CLI
自进化
Agent 框架
GitHub
```
